Необходимо провести полную проверку работоспособности эндпоинтов /api/users после последних изменений модели пользователя и API.
Проверить
GET /api/users
Убедиться, что запрос:
успешно выполняется;
возвращает HTTP 200 OK;
корректно получает список пользователей;
возвращает все необходимые поля, включая:
id;
first_name;
last_name;
telegram_id;
role;
type (если применимо);
не вызывает ошибок сериализации;
не содержит проблем с загрузкой связанных сущностей.
POST /api/users
Проверить создание пользователей для всех поддерживаемых ролей.
Создание Applicant
Проверить успешное создание пользователя с ролью:
APPLICANT
Создание Executor
Проверить создание пользователя:
role = EXECUTOR
type = SYSADMIN
и
role = EXECUTOR
type = MASTER
Убедиться, что поле type является обязательным для EXECUTOR.
Создание Admin
Проверить создание пользователя:
role = ADMIN
Проверить валидацию
Убедиться, что API корректно обрабатывает ошибки:
отсутствует обязательное поле;
отсутствует type для EXECUTOR;
передано невалидное значение type;
передана неизвестная роль;
дублирующийся telegram_id;
некорректный формат входных данных.
API должен возвращать корректные HTTP-коды и понятные сообщения об ошибках.
Проверить базу данных
После успешного создания пользователя убедиться, что:
запись появилась в БД;
все поля сохранены корректно;
role и type соответствуют отправленным значениям.
Требования
Не ограничиваться проверкой кода — выполнить реальные GET и POST запросы.
Если будут обнаружены ошибки, определить их Root Cause.
Исправить только необходимые части проекта.
Не нарушить существующую архитектуру.
Формат результата
Результаты проверки GET /api/users.
Результаты проверки POST /api/users.
Обнаруженные ошибки (если есть).
Root Cause Analysis.
Изменённые файлы.
Подтверждение, что оба эндпоинта полностью работоспособны после внесённых изменений.

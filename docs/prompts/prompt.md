Необходимо реализовать новый API-метод для обновления данных пользователя.
Требуется реализовать
Добавить endpoint для редактирования существующего пользователя.
Например:
PUT /api/users/{user_id}
Выбери наиболее подходящий вариант в соответствии с текущим API-стилем проекта.
Обновляемые поля
Необходимо предоставить возможность изменять:
first_name
last_name
telegram_id
role
type
Бизнес-правила
Role
Допустимые значения:
APPLICANT
EXECUTOR
ADMIN
Type
Для пользователей с ролью:
EXECUTOR
поле type обязательно.
Допустимые значения:
SYSADMIN
MASTER
Проверить валидацию:
если роль EXECUTOR, поле type обязательно;
если роль изменяется с EXECUTOR на другую роль, корректно обработать значение type согласно архитектуре проекта;
не допускать невалидных значений.
Валидация
Проверить:
существование пользователя;
уникальность telegram_id;
корректность роли;
корректность type;
корректность входных данных.
При ошибках возвращать соответствующие HTTP-коды и понятные сообщения.
Обновить
DTO / Schemas;
Service layer;
Repository layer;
OpenAPI / Swagger документацию;
Тесты (если используются в проекте).
Проверить
Успешные сценарии
изменение имени;
изменение фамилии;
изменение telegram_id;
изменение роли;
изменение типа EXECUTOR;
изменение нескольких полей одновременно.
Негативные сценарии
пользователь не найден;
дублирующийся telegram_id;
невалидная роль;
невалидный type;
EXECUTOR без type.
Требования
Использовать существующую архитектуру проекта.
Не дублировать бизнес-логику создания пользователя.
Переиспользовать существующие сервисы и валидаторы.
Сохранить обратную совместимость API.
Формат результата
Кратко опиши реализованный endpoint.
Перечисли изменённые файлы.
Опиши валидацию.
Приведи пример запроса и ответа.
Подтверди, что обновление пользователя корректно работает для всех поддерживаемых ролей.
